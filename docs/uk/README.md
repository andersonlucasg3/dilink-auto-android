# DiLink-Auto

Використовуйте програми свого телефону на вбудованому екрані автомобіля. З відкритим вихідним кодом, не потребує сервісів Google.

Альтернатива Android Auto з відкритим вихідним кодом для **будь-якого телефону з Android 10+** у парі з інформаційно-розважальними системами **BYD DiLink 3.0+**. Спочатку створено через відсутність підтримки Xiaomi HyperOS / китайської прошивки, але працює універсально.

## Що Він Робить

DiLink-Auto передає програми вашого телефону на дисплей автомобіля з повною сенсорною взаємодією. Запускайте навігацію, музику, повідомлення — будь-які програми на телефоні — безпосередньо з екрана автомобіля. Сповіщення з'являються на навігаційній панелі автомобіля з індикаторами прогресу. Відео H.264 з частотою до 60 кадрів/с, 8 Мбіт/с CBR, екран телефону вимкнено для економії батареї.

**Початкова мотивація:** подолання розриву, коли ваш телефон не може запускати Android Auto (китайська прошивка, немає Google Play Services), а ваш автомобіль підтримує лише Android Auto (без CarWith, CarPlay або Carlife). Але DiLink-Auto працює з будь-яким Android-телефоном — з сервісами Google чи без.

| Пристрій | Проблема |
|--------|-------|
| Xiaomi 17 Pro Max (HyperOS 3, китайська прошивка) | Немає Android Auto — немає Google Play Services |
| BYD Destroyer 05 / King (ринок Бразилії) | Лише Android Auto на головному пристрої |
| Будь-який телефон з Android 10+ | Працює незалежно від прошивки чи Play Services |

## Вимоги

**Телефон:**
- Будь-який телефон з Android 10+
- Увімкнено налагодження через USB (Параметри розробника)
- Дозвіл на доступ до всіх файлів (запитується при першому запуску)

**Автомобіль:**
- BYD DiLink 3.0 або новіше
- Один вільний порт USB-A

**Точка доступу телефону повинна бути увімкнена** — автомобіль підключається до WiFi-точки доступу вашого телефону. Без кодів сполучення, без облікового запису Google.

**Підключення до інтернету не потрібне.** DiLink-Auto передає все локально через точку доступу WiFi вашого телефону — автомобіль і телефон спілкуються безпосередньо. Підключення до інтернету необхідне лише для програм, що працюють на вашому телефоні (наприклад, навігація, музика), але не для самого DiLink-Auto.

## Як Це Працює

1. **Увімкніть точку доступу** — Увімкніть WiFi-точку доступу на телефоні. Автомобіль підключається до неї.
2. **Підключіть** — Підключіть телефон до USB-порту автомобіля
3. **Автовстановлення** — Телефон встановлює програму автомобіля через WiFi ADB (лише перший раз, один дотик)
4. **Автопідключення** — 3 виділені WiFi TCP-потоки: відео (порт 9638), сенсорне введення (порт 9639) і керування (порт 9637)
5. **Користуйтеся програмами** — Запускайте будь-які програми з лаунчера автомобіля. Програма працює на телефоні, відображається на автомобілі та реагує на дотик

Телефон запускає ваші програми на віртуальному дисплеї, кодує екран у відео H.264 і передає його на автомобіль. Дотики на екрані автомобіля відправляються назад на телефон і вводяться як реальні сенсорні події. Фізичний екран телефону залишається вимкненим (економія батареї) і може використовуватися незалежно.

## Встановлення

<a href="https://github.com/andersonlucasg3/dilink-auto-android/releases/latest"><img src="https://img.shields.io/github/v/release/andersonlucasg3/dilink-auto-android?label=Download%20Latest%20Release" alt="Download Latest Release"></a>

Завантажте останню версію або зберіть з вихідного коду:

1. **Збірка:** `./gradlew :app-client:assembleDebug`
2. **Встановіть** APK з `app-client/build/outputs/apk/debug/app-client-debug.apk` лише на ваш телефон
3. **Увімкніть налагодження через USB** на телефоні (Налаштування → Параметри розробника)
4. **Відкрийте DiLink-Auto** на телефоні та надайте доступ до всіх файлів за запитом
5. **Увімкніть точку доступу, потім підключіть до USB автомобіля** — програма автомобіля автовстановлюється при першому запуску через WiFi ADB

APK автомобіля та JAR VD-сервера вбудовані в APK телефону — вам ніколи не потрібно встановлювати щось на автомобіль самостійно.

## Поточний Статус

**Працює:**
- Потік відео H.264 60 кадрів/с (8 Мбіт/с CBR, профіль Main, налаштовується через рукостискання)
- Повне сенсорне введення (мультитач, стискання для масштабування)
- Лаунчер програм з пошуком, алфавітним сортуванням, значки 64dp, адаптивна сітка
- Сповіщення на екрані автомобіля з індикаторами прогресу, дотик для відкриття
- Самооновлення через GitHub Releases (реліз) або передрелізи (налагодження)
- **Підтримка Shizuku**: підключення без ADB, безшумне самооновлення через pm install
- Автооновлення: телефон виявляє застарілу програму автомобіля (порівняння назв версій) та оновлює її через WiFi ADB
- Екран телефону вимкнено під час передачі (економія батареї)
- Керований onboarding для всіх необхідних дозволів
- Інтернаціоналізація: англійська, португальська, російська, білоруська, французька, казахська, українська, узбецька
- Відновлення дисплея після відключення USB (v0.14.0+)
- Протестовано на BYD DiLink 3.0 (1920x990) + Xiaomi 17 Pro Max (Android 16) + POCO F5

**Незабаром:** аудіопотік, керування медіа, віджети навігації

**Відомі обмеження:**
- Процес VD-сервера перезапускається при відключенні USB (перепідключається автоматично).
- Точку доступу потрібно вмикати вручну (обмеження Android 16).
- Часом візуальні артефакти — гонка перезапуску декодера, відновлюється на наступному ключовому кадрі (~1 с).
- Затримка передачі ~100-200 мс під навантаженням. CBR 8 Мбіт/с.
- Дисплей може залишатися вимкненим після різкого відключення USB (виправлено у v0.14.0).
- **Деякі застосунки не заповнюють екран (letterbox/лише портрет).** DiLink-Auto дзеркально відображає віртуальний дисплей в альбомній орієнтації на екран автомобіля. Застосунки, які не підтримують альбомну орієнтацію, відображатимуться зі смугами або вузькими — це повністю контролюється кожним застосунком, а не DiLink-Auto. Нічого не можна зробити з боку дзеркалювання.

## Структура Проєкту

APK телефону (`app-client`) містить як APK автомобіля (`app-server`), так і JAR VD-сервера. Коли ви встановлюєте програму на телефон, все необхідне вже всередині.

```
DiLink-Auto/
├── protocol/       Спільна бібліотека (кадрування, повідомлення, виявлення, USB ADB)
├── app-client/     APK телефону — ретранслятор, розгортання VD, автооновлення автомобіля, FileLog
├── app-server/     APK автомобіля — UI, стан машини з'єднання, відеодекодер
├── vd-server/      Сервер VirtualDisplay (скомпільовано в JAR, розгортається телефоном)
├── docs/           Документація
└── gradle/         Система збірки
```

## Участь

PR вітаються. Дивіться [Архітектура](./architecture.md) та [Протокол](./protocol.md) для технічного контексту. Збірка з `./gradlew :app-client:assembleDebug` (JDK 17+, Android SDK 34).

### Модель Галуження (Git-Flow + Типи Завдань)

Гілки створюються автоматично агентом завдань на основі використаного **шаблону завдання**:

| Шаблон | Позначка | Зразок Гілки | Призначення |
|----------|-------|---------------|---------|
| Виправлення Помилки | `bug` | `fix/N-agent` | Виправлення помилок |
| Нова Функція | `feature` | `feature/N-agent` | Нові функції |
| Дослідження | `investigation` | `investigate/N-agent` | Дослідження кодової бази |
| Документація | `documentation` | `docs/N-agent` | Оновлення документації |
| Реліз | `release` | `release/vX.Y.Z` | Підготовка релізу |
| Завдання Агента (загальне) | — | `issue/N-agent` | Загальне |

Усі гілки зливаються в `develop` через PR, крім `release/*`, яка спрямовується на `main`.

### CI Процеси

| Процес | Тригер | Дія |
|----------|---------|--------|
| `build.yml` | Push/PR у `main` | Перевірка: збірка релізного APK |
| `build-develop.yml` | Push/PR у `develop`, `release/*` | Перевірка: збірка налагоджувального APK |
| `build-pre-release.yml` | Тег `vX.Y.Z-dev-NN` | Збірка налагоджувального APK + GitHub передреліз |
| `build-release.yml` | Тег `vX.Y.Z` | Збірка підписаного релізного APK + GitHub Release |
| `sync-main-to-develop.yml` | Push у `main` | Злиття `main` → `develop` (зворотна синхронізація git-flow) |
| `issue-agent.yml` | Завдання відкрито / коментар | Автономний агент: гілка, збірка, PR |

Весь CI працює на **самостійно розміщених WSL раннерах**.

**Процес релізу:** Створіть завдання Release із шаблону. Агент створює `release/vX.Y.Z`, готує зміни та позначає тегом `vX.Y.Z-dev-NN`. Відправка гілки релізу запускає `build-pre-release.yml`, який знаходить тег `-dev` на коміті через `git tag --points-at HEAD` і публікує передреліз. Коли готово, `release/vX.Y.Z` зливається в `main` із тегом `vX.Y.Z` на коміті злиття. Push у `main` запускає `build-release.yml` (збірка підписаного APK + створення GitHub Release) і `sync-main-to-develop.yml` (автоматичне злиття `main` назад у `develop`).

**Оновлення передрелізів:** Користувачі на каналі передрелізів отримують збірки `-dev`. Користувачі на каналі релізів отримують лише стабільні збірки. Канал налаштовується в Налаштуваннях.
