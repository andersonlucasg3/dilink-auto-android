# Серверний додаток автомобіля (app-server)

## Огляд

Серверний додаток автомобіля працює на інформаційно-розважальній системі BYD DiLink. Він використовує **паралельну модель з'єднання** з одночасною роботою треків WiFi (3 виділені з'єднання) та USB:

1. Track A (WiFi): gateway IP + mDNS discovery, control connect (9637), handshake з телефоном
2. Після handshake: відео (9638) + вхідні (9639) з'єднання відкриваються паралельно
3. Track B (USB): сканування пристроїв, USB ADB connect (з logSink діагностикою), запуск додатку телефону
4. `checkAndAdvance()` оцінює стан при зміні будь-якої передумови
5. Отримує H.264 відео та рендерить на дисплеї автомобіля (ранній старт декодера на позаекранній поверхні)
6. Захоплює події дотику та надсилає на телефон через вхідне з'єднання для ін'єкції вводу VD

Стани: IDLE → CONNECTING → CONNECTED → STREAMING

APK автомобіля вбудований в APK телефону. Телефон автоматично оновлює додаток автомобіля через dadb при виявленні невідповідності версій під час handshake. Автомобіль отримує повідомлення `UPDATING_CAR` і показує статус замість сліпого перепідключення.

### Дворежимний UI

Додаток автомобіля розділяє потік з'єднання від потокового досвіду на два окремі режими, відповідаючи підходу додатку телефону:

- **Launch mode** (`CarLaunchScreen`): Повноекранний, орієнтований на з'єднання. Показує брендинг, покрокові інструкції зі з'єднання, статус з'єднання та ручне введення IP. Немає панелі навігації — весь екран присвячений встановленню з'єднання. Показується при запуску додатку автомобіля і залишається, поки іконки додатків не будуть отримані від телефону через control-з'єднання.

- **Streaming mode** (`CarShell` з `PersistentNavBar`): Знайомий макет з лівою панеллю навігації (сповіщення, додому, назад, нещодавні додатки) та областю вмісту (сітка додатків, дзеркальний вигляд, сповіщення). Показується, коли `appList` не порожній і стан CONNECTED або STREAMING.

Тригер переходу: коли телефон надсилає `APP_LIST` через control-з'єднання і стан з'єднання досягає CONNECTED/STREAMING, UI перемикається з режиму launch на режим streaming.

## Компоненти

### CarConnectionService

Сервіс переднього плану, який керує повним життєвим циклом з'єднання з паралельним скінченним автоматом передумов та 3 виділеними з'єднаннями.

**Архітектура 3 з'єднань:**
- `controlConnection`: handshake, heartbeat, команди додатків, канал DATA (список додатків, сповіщення, логи автомобіля)
- `videoConnection`: лише відеокадри H.264 (phone → car)
- `inputConnection`: лише події дотику (car → phone)
- Heartbeat/watchdog лише на control-з'єднанні; відео та вхідні не мають накладних витрат heartbeat
- Смерть будь-якого з'єднання каскадно → повний розрив сесії

**Архітектура паралельних треків:**
- `connectionScope`: батьківський Job для всіх корутин discovery, скасовується при відключенні
- Track A та Track B працюють одночасно
- `checkAndAdvance()` оцінює загальний стан при зміні будь-якої передумови
- Відключення користувача: залишається IDLE, без автоперепідключення (зберігається в SharedPreferences)

**Track A — WiFi:**
- Discovery: gateway IP (hotspot/LAN, повторюється кожні 3s) → mDNS → manual IP
- Control connection: NIO неблокуючий SocketChannel connect до TCP телефону:9637
- Handshake: надсилає розміри вікна + DPI + appVersionCode + targetFps (60) → отримує інформацію про телефон + vdServerJarPath
- При відповіді handshake: відкриває відео (9638) + вхідні (9639) з'єднання паралельно, встановлює `wifiReady = true` після встановлення всіх 3
- Video: отримує кадри H.264 через відео-з'єднання, відправляє до VideoDecoder
- Touch: виділений однопотоковий виконавець, `sendTouchEvent()` / `sendTouchBatch()` через вхідне з'єднання
- Heartbeat: інтервал 3s, тайм-аут watchdog 10s (лише control-з'єднання)
- Backoff: експоненційна затримка при збоях перепідключення

**Track B — USB:**
- Реєструє `BroadcastReceiver` для `USB_DEVICE_ATTACHED` / `USB_DEVICE_DETACHED`
- `MainActivity` пересилає USB intent до сервісу
- Сканує USB-пристрої на інтерфейс ADB
- USB ADB connect через UsbAdbConnection (у модулі protocol/), з `logSink`, який маршрутизує всі логи автентифікації ADB до FileLog телефону
- Запускає додаток телефону: `am start -n com.dilinkauto.client/.MainActivity`

**Потік оновлення:**
- Якщо телефон надсилає `UPDATING_CAR`, автомобіль встановлює `updatingFromPhone = true`, показує "Updating car app..." статус
- Пропускає спроби відео/вхідних з'єднань та цикл перепідключення під час оновлення
- Після `pm install -r`, додаток автомобіля перезапускається наново

**State Flows:**
- `_state`, `_phoneName`, `_appList`, `_notifications`, `_mediaMetadata`, `_playbackState`: основний стан для UI
- `_videoReady`: true, коли надходить перший неконфігураційний відеокадр
- `_statusMessage`: читабельний статус для відображення в UI
- `_vdStackEmpty` (SharedFlow): випромінюється, коли телефон повідомляє, що VD не має activity (запускає навігацію додому)

**Запуск VD-сервера:**
- `deployVdServer()`: `shellNoWait` з CLASSPATH з vdServerJarPath handshake
- Args: `W H DPI PORT EW EH FPS` — FPS передається з targetFps handshake

**Ранній старт декодера:**
- При першому відеокадрі CONFIG запускає VideoDecoder на позаекранному SurfaceTexture (перед MirrorScreen)
- `onSurfaceTextureAvailable` MirrorScreen зупиняє та перезапускає декодер з реальною поверхнею TextureView
- Запобігає втраті ключових кадрів під час затримки компонування UI

**Маршрутизація логів автомобіля:**
- `carLogSend()` маршрутизує всі логи з боку автомобіля через DATA-канал `CAR_LOG` на телефон
- `videoDecoder.logSink` та `adb.setLogSink()` з'єднують логи VideoDecoder та UsbAdbConnection через той самий шлях
- Усі логи видимі в `/sdcard/DiLinkAuto/client.log` телефону

### VideoDecoder

Декодер H.264 з використанням MediaCodec з виводом на Surface (прямий рендеринг GPU).

- Черга кадрів: 15 кадрів — буферизує гонку старту та мережевий джиттер
- `onFrameReceived()`: ставить кадри в чергу навіть до виклику `start()`
- `start()`: спочатку подає кешований CONFIG, потім спустошує чергу
- Видалення найстарішого при заповненій черзі: віддає перевагу видаленню P-кадрів, витісняє P-кадри в черзі для ключових кадрів/CONFIG
- `KEY_LOW_LATENCY = 1`, `KEY_PRIORITY = 0` для мінімальної затримки декодування
- CONFIG (SPS/PPS) кешується та переграється при перезапуску декодера
- `isRunning` для координації раннього старту
- Зворотний виклик `logSink` маршрутизує всі логи декодера на телефон через carLogSend
- Режим catchup: чотири зони прискорення на основі глибини черги — normal (0-6 кадрів), gentle 1.5x (7-12 кадрів, пропускає 1 з 3 неключових), medium 2x (13-20 кадрів, пропускає 1 з 2), aggressive 3x (21+ кадрів, пропускає 2 з 3). Ключові кадри ніколи не пропускаються.
- Очищає кодек і повторно подає CONFIG при 10+ послідовних скидах dequeueInputBuffer

### ServerApp

Клас Application. Створює канал сповіщень `dilinkauto_car_service` з `IMPORTANCE_LOW`.

### RemoteAdbController

Прямий клієнт ADB з використанням бібліотеки dadb. Надає tap, swipe, back, home та запуск додатків через shell-команди на віртуальному дисплеї. Використовується як альтернативний шлях вводу.

### CarLaunchScreen

Повноекранний компонувальний елемент, орієнтований на з'єднання, який показується до встановлення з'єднання з телефоном — без панелі навігації, без сітки додатків.

- Брендинг DiLink Auto (іконка, заголовок, слоган)
- Картка статусу з'єднання з кольоровим індикатором (зелений=streaming, помаранчевий=connected/connecting, сірий=idle) та живим текстом статусу
- Інструкції "How to connect": 4 нумеровані кроки (увімкнути hotspot, підключити USB, відкрити додаток телефону, чекати автопідключення)
- Ручне введення IP для прямого з'єднання
- Замінюється макетом режиму streaming, коли `appList` стає непустим і стан досягає CONNECTED/STREAMING

### PersistentNavBar

76dp ліва панель навігації — **показується лише в режимі streaming** — з:
- Годинник (HH:mm, оновлюється кожну 1s)
- Кнопка Eject (відключає та зберігає перевагу користувача)
- Індикатор стану мережі
- Кнопка сповіщень з лічильником непрочитаних
- Кнопка Home
- Кнопка Back
- Іконки нещодавніх додатків (макс. 5, очищуються, коли додатки стають недоступними)
- Іконки 40dp, текст 12-14sp

Ширина обчислюється для гарантії парного вікна для кодувальника H.264.

### NotificationScreen

- Список сповіщень, відсортований за часом (спочатку нові)
- Дедублікація за ID: оновлення замінюють існуючі (обробляє сповіщення з прогресом)
- Смуги прогресу: determinate (заповнені) та indeterminate (обертові)
- Запуск по дотику: дотик до сповіщення запускає додаток-власник на VD і перемикає на дзеркальний вигляд

### Сітка додатків (HomeContent)

Показується як основна область вмісту, коли режим streaming активний і поточний екран HOME:
- Поле пошуку з `imePadding()` — клавіатура не штовхає activity, рухається лише рядок пошуку
- `windowSoftInputMode="adjustNothing"` у manifest
- Іконки додатків 64dp в адаптивних комірках сітки 160dp
- Текст назви додатку: bodyLarge
- Алфавітне сортування
- Ручне введення IP
- Статус з'єднання

### LauncherScreen (Застарілий)

Повний інтегрований макет launcher з `CarStatusBar`, `SideNavBar` (80dp) та `AppGrid`. Не використовується в поточній маршрутизації `CarShell` — активний UI використовує `PersistentNavBar` + `HomeContent`/`MirrorContent`/`NotificationContent` вбудовані компонувальні елементи.

### RecentAppsState

Відстежує нещодавно запущені додатки (макс. 5), зберігається в SharedPreferences. `pruneUnavailable()` видаляє додатки, яких більше немає при оновленні списку додатків.

### NavBarComponents

Окремі компонувальні елементи віджетів панелі навігації: `ClockDisplay` (оновлюється кожну 1s), `NetworkInfo` (стан connected/disconnected), `RecentAppIcon` (40dp, з підсвіткою активного стану), `NavActionButton` (іконки 40dp, мітки 12sp).

### CarTheme

Темна колірна схема Material3 (`CarDark`) з категорійними кольорами плиток додатків: Navigation (зелений), Music (рожевий), Communication (синій), Other (сірий).

### Інформація про дисплей автомобіля

Протестовано на BYD DiLink 3.0:
- Екран: 1920x990 @ 240dpi
- Вікно: ~1806x990 (після 76dp панелі навігації)
- VD: ~3282x1800 @ 480dpi (GPU-масштабовано до ~1806x990 для кодування)

## Залежності

- Jetpack Compose + Material 3
- Protocol module (спільний з додатком телефону, включає UsbAdbConnection + AdbProtocol + VideoConfig)
- dadb 1.2.10 (TCP ADB резервний варіант)
