# Клиентское приложение телефона (app-client)

## Обзор

Клиент телефона управляет развёртыванием сервера VD, автообновлением автомобиля и ретрансляцией по 3 соединениям. Приложение телефона:

1. Слушает TCP соединения от автомобиля на порту 9637 (управление, NIO ServerSocketChannel)
2. Отвечает на handshake информацией об устройстве, vdServerJarPath и читает `targetFps`
3. Проверяет appVersionCode из handshake — отправляет `UPDATING_CAR` и автообновляет приложение авто через dadb при несовпадении версий
4. Принимает видео (9638) и ввод (9639) соединения от автомобиля после handshake
5. Развёртывает vd-server.jar в `/sdcard/DiLinkAuto/` и запускает сервер VD (с аргументом FPS)
6. Принимает обратное соединение от сервера VD на localhost:19637 (NIO ServerSocketChannel)
7. Ретранслирует видео H.264 от сервера VD к автомобилю через видео соединение
8. Ретранслирует события касаний от автомобиля (соединение ввода) к серверу VD, с диспетчеризацией на `Dispatchers.IO`

**Никакого захвата экрана. Никакого MediaProjection.** Всё видео поступает от процесса сервера VD.

## Компоненты

### ClientApp

Класс приложения. Создаёт каналы уведомлений (`dilinkauto_service`, `dilinkauto_update`), инициализирует `UpdateManager` при создании.

### UpdateManager

Механизм самообновления, который проверяет GitHub Releases на новые версии.
- `checkForUpdate(force)`: Запрашивает `https://api.github.com/repos/andersonlucasg3/dilink-auto-android/releases/latest`, сравнивает semver из имени тега с установленной versionName. Соблюдает 6-часовой интервал, если не принудительно.
- `downloadUpdate()`: Скачивает APK через `HttpsURLConnection` с отчётом о прогрессе. Проверяет через `PackageManager.getPackageArchiveInfo()`.
- `installUpdate(context)`: Открывает системный установщик пакетов через URI `FileProvider`.
- Состояния: Idle, Checking, Available, Downloading, ReadyToInstall, UpToDate, Error. Предоставляется через `StateFlow`.

### MainActivity

Точка входа с двумя экранами:

- **OnboardingScreen** (первый запуск): Мастер из 7 шагов — Приветствие, Доступ ко всем файлам, Оптимизация батареи, Служба специальных возможностей, Доступ к уведомлениям, Настройка авто, Готово. Каждый шаг разрешения объясняет, что не работает без него. Автоматически продвигается при предоставлении разрешения. Пользователь может пропустить любой шаг.
- **ClientScreen** (последующие запуски): карточка статуса, кнопка старт/стоп, Установить на автомобиль, карточка самообновления, кнопка Поделиться логами и статус оставшихся разрешений для пропущенных при онбординге.

### ConnectionService

Служба переднего плана, управляющая жизненным циклом соединения телефон-авто с 3 выделенными соединениями. Автозапускается при открытии приложения телефона (например, через USB ADB автомобиля).

- **Управляющее соединение** (порт 9637): NIO TCP сервер на `0.0.0.0`, обрабатывает handshake, heartbeat, команды приложений, канал DATA
- **Видео соединение** (порт 9638): принимается после handshake, передаётся VirtualDisplayClient для ретрансляции видео
- **Соединение ввода** (порт 9639): принимается после handshake, слушатель кадров INPUT с диспетчеризацией на `Dispatchers.IO` для избежания NetworkOnMainThreadException при записи касаний на localhost
- `deployAssets()`: извлекает vd-server.jar на sdcard, app-server.apk в filesDir
- Определяет несовпадение версий → отправляет `UPDATING_CAR` → автообновляет приложение авто через dadb (WiFi ADB, dadb 1.2.10)
- Умный сетевой обратный вызов: `onLost` проверяет, является ли потерянная сеть той, которую использует соединение, игнорирует несвязанные потери (циклирование мобильных данных)
- Ретранслирует видеокадры (H.264 CONFIG + FRAME) от VD к авто через видео соединение
- Направляет события касаний от авто (соединение ввода) к серверу VD
- Отправляет список приложений с иконками PNG 96x96 в авто через управляющее соединение
- Обрабатывает команды LAUNCH_APP, GO_BACK, GO_HOME и пересылает их серверу VD
- Ставит в очередь запуск приложений, если сервер VD ещё не подключён
- Регистрирует сервис mDNS для автообнаружения автомобилем
- `FileLog.rotate()` при старте службы — архивирует лог предыдущей сессии
- Кнопка "Установить на автомобиль": ручная установка + автоматическая при несовпадении версий при handshake

### VirtualDisplayClient

Принимает обратное соединение от процесса сервера VD на `localhost:19637`. Принимает два параметра Connection: `videoConnection` и `controlConnection`.

- NIO ServerSocketChannel accept (неблокирующий) — сервер VD подключается К телефону
- Читает: `MSG_VIDEO_CONFIG`, `MSG_VIDEO_FRAME`, `MSG_STACK_EMPTY`, `MSG_DISPLAY_READY`
- Пишет: `CMD_LAUNCH_APP`, `CMD_GO_BACK`, `CMD_GO_HOME`, `CMD_INPUT_TOUCH` (0x32)
- Видеокадры ретранслируются через `videoConnection.sendVideo()` (изолированы от трафика управления)
- Сигнал пустого стека (`MSG_STACK_EMPTY`) пересылается в авто через `controlConnection.sendControl()`
- Записи касаний на localhost синхронны с `FrameCodec.writeAll()` под `writeLock`
- При отключении: восстанавливает физический экран (`cmd display power-on 0` + `KEYCODE_WAKEUP`) как страховку при завершении процесса сервера VD до очистки

### AdbBridge

Резервный помощник shell команд. Предоставляет `execShell()` и `execFast()` с использованием `Runtime.exec()` для операций сервера VD и управления питанием экрана при сбое прямой рефлексии API.

### VirtualDisplayManager

Управляет запуском приложений на физическом экране, когда VD не используется. Связывается с `InputInjectionService` для инжекции ввода на основе жестов.

### VideoEncoder

Кодер H.264 MediaProjection + MediaCodec с использованием виртуального дисплея `AUTO_MIRROR`. Альтернативный путь кодирования (не используется в основном конвейере стриминга, идущем через сервер VD).

### FileLog

Файловый регистратор, обходящий фильтрацию logcat Android (HyperOS фильтрует `Log.i/d` для несистемных приложений).

- Пишет в `/sdcard/DiLinkAuto/client.log`
- `rotate()`: архивирует текущий лог как `client-YYYYMMDD-HHmmss.log`, начинает новый
- `zipLogs()`: создаёт `dilinkauto-logs.zip` из всех файлов `.log` для отправки
- Хранит максимум 10 логов (9 архивных + текущий)
- Потокобезопасный: ConcurrentLinkedQueue без блокировок, опустошаемая потоком записи
- Также вызывает `android.util.Log.*` для стандартного вывода logcat

### Ретрансляция Multi-Touch

События касаний поступают от автомобиля через соединение ввода как CMD_INPUT_TOUCH (0x32) с сырыми данными MotionEvent. `handleInputFrame` диспетчеризуется на `Dispatchers.IO` (не Main) для возможности записи в сокет localhost. Телефон передаёт события DOWN/MOVE/UP с pointerId напрямую серверу VD, который обрабатывает полное построение MotionEvent со всеми активными указателями.

## Необходимые разрешения

| Разрешение | Назначение |
|-----------|---------|
| MANAGE_EXTERNAL_STORAGE | Доступ ко всем файлам для развёртывания JAR VD на sdcard |
| Служба специальных возможностей | Инжекция ввода на физический экран (резервный вариант) |
| Доступ к уведомлениям | Пересылка уведомлений в авто (с прогрессом) |
| USB Debugging | Требуется для дорожки USB ADB автомобиля (Опции разработчика) |

## Зависимости

- Jetpack Compose + Material 3
- kotlinx-coroutines
- dadb 1.2.10 (WiFi ADB для автообновления авто)
- Модуль Protocol (общий с приложением автомобиля)
