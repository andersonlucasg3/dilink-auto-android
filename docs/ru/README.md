# DiLink-Auto

Используйте приложения вашего телефона на встроенном экране автомобиля. С открытым исходным кодом, без необходимости Google Services.

Альтернатива Android Auto с открытым исходным кодом для **любого телефона на Android 10+** в паре с информационно-развлекательными системами **BYD DiLink 3.0+**. Изначально проект был создан для устранения разрыва между Xiaomi HyperOS / китайской прошивкой, но работает универсально.

## Что он делает

DiLink-Auto зеркалирует приложения вашего телефона на дисплей автомобиля с полноценным сенсорным взаимодействием. Запускайте навигацию, музыку, сообщения — любое приложение на телефоне — прямо с экрана автомобиля. Уведомления отображаются на панели навигации автомобиля с индикаторами прогресса. Видео H.264 с частотой до 60 кадров/с, CBR 8 Мбит/с, с выключенным экраном телефона для экономии заряда батареи.

**Изначальная мотивация:** устранение разрыва, когда ваш телефон не может запустить Android Auto (китайская прошивка, нет Google Play Services), но автомобиль поддерживает только Android Auto (нет CarWith, CarPlay или Carlife). Однако DiLink-Auto работает с любым телефоном на Android — с Google Services или без них.

| Устройство | Проблема |
|--------|-------|
| Xiaomi 17 Pro Max (HyperOS 3, китайская прошивка) | Нет Android Auto — нет Google Play Services |
| BYD Destroyer 05 / King (бразильский рынок) | Только Android Auto на головном устройстве |
| Любой телефон на Android 10+ | Работает независимо от прошивки или Play Services |

## Требования

**Телефон:**
- Любой телефон на Android 10+
- Включена отладка по USB (Параметры разработчика)
- Разрешение «Доступ ко всем файлам» (запрашивается при первом запуске)

**Автомобиль:**
- BYD DiLink 3.0 или новее
- Один свободный порт USB-A

**Точка доступа телефона должна быть включена** — автомобиль подключается к WiFi-точке доступа вашего телефона. Никаких кодов сопряжения, учётная запись Google не требуется.

**Подключение к интернету не требуется.** DiLink-Auto передаёт всё локально через точку доступа WiFi вашего телефона — автомобиль и телефон общаются напрямую. Подключение к интернету необходимо только для приложений, работающих на вашем телефоне (например, навигация, музыка), но не для самого DiLink-Auto.

## Как это работает

1. **Включите точку доступа** — Включите WiFi-точку доступа на телефоне. Автомобиль подключается к ней.
2. **Подключите кабель** — Подключите телефон к USB-порту автомобиля
3. **Автоустановка** — Телефон устанавливает приложение на автомобиль через WiFi ADB (только при первом подключении, одно нажатие)
4. **Автоподключение** — 3 выделенных WiFi TCP-потока: видео (порт 9638), сенсорный ввод (порт 9639) и управление (порт 9637)
5. **Используйте свои приложения** — Запускайте любое приложение с экрана запуска автомобиля. Оно работает на телефоне, отображается на экране автомобиля и реагирует на касания

Телефон запускает ваши приложения на виртуальном дисплее, кодирует экран в видео H.264 и передаёт его на автомобиль. Касания экрана автомобиля отправляются обратно на телефон и внедряются как реальные события касания. Физический экран телефона остаётся выключенным (экономия заряда) и может использоваться независимо.

## Установка

<a href="https://github.com/andersonlucasg3/dilink-auto-android/releases/latest"><img src="https://img.shields.io/github/v/release/andersonlucasg3/dilink-auto-android?label=Download%20Latest%20Release" alt="Download Latest Release"></a>

Загрузите последнюю версию или соберите из исходного кода:

1. **Сборка:** `./gradlew :app-client:assembleDebug`
2. **Установите** APK из `app-client/build/outputs/apk/debug/app-client-debug.apk` только на ваш телефон
3. **Включите отладку по USB** на телефоне (Настройки → Параметры разработчика)
4. **Откройте DiLink-Auto** на телефоне и предоставьте разрешение «Доступ ко всем файлам» по запросу
5. **Включите точку доступа, затем подключите телефон к USB автомобиля** — приложение для автомобиля автоматически установится при первом запуске через WiFi ADB

APK для автомобиля и JAR VD-сервера упакованы внутри APK телефона — вам никогда не нужно устанавливать что-либо на автомобиль самостоятельно.

## Текущее состояние

**Работает:**
- Потоковое видео H.264 60 кадров/с (CBR 8 Мбит/с, профиль Main, настраивается через рукопожатие)
- Полноценный сенсорный ввод (мультитач, масштабирование щипком)
- Пусковая панель приложений с поиском, алфавитной сортировкой, иконками 64dp, адаптивная сетка
- Уведомления на экране автомобиля с индикаторами прогресса, нажмите чтобы открыть
- Самообновление через GitHub Releases (релиз) или предварительные выпуски (отладка)
- **Поддержка Shizuku**: подключение без ADB, бесшумное самообновление через pm install
- Автообновление: телефон обнаруживает устаревшее приложение на автомобиле (сравнение имён версий) и обновляет его через WiFi ADB
- Экран телефона выключен во время передачи (экономия заряда)
- Пошаговое руководство по настройке всех необходимых разрешений
- Интернационализация: английский, португальский, русский, белорусский, французский, казахский, украинский, узбекский
- Восстановление дисплея после отключения USB (v0.14.0+)
- Протестировано на BYD DiLink 3.0 (1920x990) + Xiaomi 17 Pro Max (Android 16) + POCO F5

**В разработке:** потоковая передача аудио, элементы управления мультимедиа, виджеты навигации

**Известные ограничения:**
- Процесс VD-сервера перезапускается при отключении USB (переподключается автоматически).
- Точку доступа необходимо включать вручную (ограничение Android 16).
- Иногда возникают визуальные артефакты — гонка перезапуска декодера, восстанавливается на следующем ключевом кадре (~1 с).
- Задержка передачи ~100-200 мс под нагрузкой. CBR 8 Мбит/с.
- Дисплей может оставаться выключенным после резкого отключения USB (исправлено в v0.14.0).
- **Некоторые приложения не заполняют экран (letterbox/только портрет).** DiLink-Auto зеркально отображает виртуальный дисплей в альбомной ориентации на экран автомобиля. Приложения, не поддерживающие альбомную ориентацию, будут отображаться с полосами или узкими — это полностью контролируется каждым приложением, а не DiLink-Auto. Со стороны зеркалирования ничего сделать нельзя.

## Структура проекта

APK телефона (`app-client`) содержит в себе как APK автомобиля (`app-server`), так и JAR VD-сервера. При установке приложения на телефон всё необходимое уже упаковано внутри.

```
DiLink-Auto/
├── protocol/       Общая библиотека (кадрирование, сообщения, обнаружение, USB ADB)
├── app-client/     APK телефона — ретрансляция, развёртывание VD, автообновление автомобиля, FileLog
├── app-server/     APK автомобиля — интерфейс, конечный автомат подключения, видеодекодер
├── vd-server/      Сервер VirtualDisplay (компилируется в JAR, развёртывается телефоном)
├── docs/           Документация
└── gradle/         Сборочная система
```

### Модель ветвления (Git-Flow + Типы задач)

Ветки создаются автоматически агентом задач на основе используемого **шаблона задачи**:

| Шаблон | Метка | Шаблон ветки | Назначение |
|----------|-------|---------------|---------|
| Исправление ошибки | `bug` | `fix/N-agent` | Исправления ошибок |
| Новая функция | `feature` | `feature/N-agent` | Новые функции |
| Исследование | `investigation` | `investigate/N-agent` | Исследование кодовой базы |
| Документация | `documentation` | `docs/N-agent` | Обновления документации |
| Релиз | `release` | `release/vX.Y.Z` | Подготовка релиза |
| Задача агента (общая) | — | `issue/N-agent` | Сборная категория |

Все ветки сливаются в `develop` через PR, кроме `release/*`, которая нацелена на `main`.

### CI рабочие процессы

| Рабочий процесс | Триггер | Действие |
|----------|---------|--------|
| `build.yml` | Push/PR в `main` | Проверка: сборка релизного APK |
| `build-develop.yml` | Push/PR в `develop`, `release/*` | Проверка: сборка отладочного APK |
| `build-pre-release.yml` | Тег `vX.Y.Z-dev-NN` | Сборка отладочного APK + GitHub предварительный релиз |
| `build-release.yml` | Тег `vX.Y.Z` | Сборка подписанного релизного APK + GitHub Release |
| `sync-main-to-develop.yml` | Push в `main` | Слияние `main` → `develop` (обратная синхронизация git-flow) |
| `issue-agent.yml` | Задача открыта / комментарий | Автономный агент: ветка, сборка, PR |

Все CI запускаются на **собственных WSL раннерах**.

**Процесс релиза:** Создайте задачу Release из шаблона. Агент создаёт `release/vX.Y.Z`, подготавливает изменения и помечает тегом `vX.Y.Z-dev-NN`. Пуш релизной ветки запускает `build-pre-release.yml`, который находит `-dev` тег на коммите через `git tag --points-at HEAD` и публикует предварительный релиз. Когда всё готово, `release/vX.Y.Z` сливается в `main` с тегом `vX.Y.Z` на коммите слияния. Пуш в `main` запускает `build-release.yml` (сборка подписанного APK + создание GitHub Release) и `sync-main-to-develop.yml` (автоматическое слияние `main` обратно в `develop`).

**Обновления предварительных релизов:** Пользователи на канале Pre-release получают `-dev` сборки. Пользователи на канале Release получают только стабильные сборки. Канал настраивается в Настройках.
