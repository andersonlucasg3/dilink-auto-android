### v0.13.0 — Исправление аутентификации USB ADB (2026-04-25)

Найдена и исправлена корневая причина: `Signature.getInstance("SHA1withRSA")` дважды хеширует AUTH_TOKEN ADB. 20-байтовый токен ADB — это предварительно хешированное значение — `RSA_sign(NID_sha1)` из AOSP рассматривает его как уже хешированное. Теперь используется `NONEwithRSA` с вручную добавленным префиксом SHA-1 DigestInfo ASN.1 (предварительно хешированная подпись). "Всегда разрешать" теперь сохраняется корректно — AUTH_SIGNATURE принимается при повторном подключении без диалога.

### v0.13.0 — Питание экрана + Кодирование ключа (2026-04-25)

- **Питание экрана через SurfaceControl (Android 14+)**: Загружает `DisplayControl` из `/system/framework/services.jar` через `ClassLoaderFactory.createClassLoader()` + нативную библиотеку `android_servers`. Резервный вариант `cmd display power-off/on` при сбое рефлексии.
- **Восстановление экрана при отключении**: `VirtualDisplayClient.disconnect()` телефона выполняет `cmd display power-on 0` + `KEYCODE_WAKEUP` как страховку при завершении процесса сервера VD до очистки.
- **Переписывание кодирования ключа ADB**: Переписана `encodePublicKey()` в точном соответствии с эталоном AOSP — исправлены константы, явный `bigIntToLEPadded()`, логирование заголовка структуры.
- **Catchup декодера**: Когда очередь превышает `100ms * TARGET_FPS / 1000` кадров (6 при 60fps), пропускает каждый второй неключевой кадр. Изображение движется с 2x скоростью, постепенно догоняя без скачков.
- **Буфер логов авто**: 200 → 10 000 сообщений. Логи аутентификации USB ADB теперь сохраняются до тех пор, пока управляющее соединение не сольёт их.

### v0.10.0 — Архитектура 3 соединений (2026-04-24)

Разделение одного мультиплексированного TCP соединения на 3 выделенных для устранения межканальных помех, вызывающих задержки видео:
- **Управляющее соединение** (порт 9637): handshake, heartbeat, команды приложений, канал DATA
- **Видео соединение** (порт 9638): только H.264 CONFIG + FRAME (телефон → авто)
- **Соединение ввода** (порт 9639): только события касаний (авто → телефон)

Каждое соединение имеет собственный экземпляр `Connection` с независимыми SocketChannel, NioReader и очередью записи. Heartbeat/watchdog только на управлении.

### v0.9.2 — Диагностическая сборка (2026-04-23)

Комплексное логирование для исследования зависания видеокадра после ~420 кадров:
- **Цикл ретрансляции видео**: логи до/после readByte, размер нагрузки, неизвестные msgTypes
- **NioReader**: логи при channel.read() возвращает 0 (каждое 100-е событие с состоянием буфера)

## Поток соединения

```
1. Приложение телефона запускается → развёртывает JAR VD, ротация FileLog, запрос исключения батареи
2. Телефон подключён к авто USB → авто обнаруживает USB_DEVICE_ATTACHED
3. Дорожка A (WiFi): авто обнаруживает телефон через IP шлюза (повтор 3с) или mDNS
4. Дорожка B (USB): авто подключает USB ADB (диагностика logSink), запускает приложение телефона
5. Управление (9637): TCP подключение → handshake (viewport + DPI + версия + targetFps)
6. Телефон: проверяет версию → при несовпадении отправляет UPDATING_CAR → автообновление через dadb
7. Видео (9638) + Ввод (9639): авто подключает параллельно после handshake
8. Телефон: принимает оба, открывает VD ServerSocket на localhost:19637
9. USB: авто запускает сервер VD (shellNoWait + exec app_process, FPS как аргумент)
10. Сервер VD: VD + SurfaceScaler (периодическая перерисовка) + кодер → NIO подключение localhost:19637
11. Авто: запускает VideoDecoder на внеэкранной поверхности при первом CONFIG кадре
12. MirrorScreen отображается → декодер перезапускается с реальной поверхностью TextureView
13. Видео: VD → SurfaceScaler → кодер → очередь записи NIO → localhost → NioReader телефона → videoConnection → WiFi TCP → NioReader авто → VideoDecoder → TextureView
14. Касания: авто TextureView → inputConnection → WiFi TCP → телефон (Dispatchers.IO) → NIO Selector сервера VD → инжекция IInputManager
15. Логи авто: carLogSend() + обратные вызовы logSink → DATA CAR_LOG → FileLog телефона
```
