# Серверное приложение автомобиля (app-server)

## Обзор

Серверное приложение автомобиля работает на информационно-развлекательной системе BYD DiLink. Оно использует **параллельную модель соединения** с WiFi (3 выделенных соединения) и USB дорожками, работающими одновременно:

1. Дорожка A (WiFi): IP шлюза + обнаружение mDNS, управляющее соединение (9637), handshake с телефоном
2. После handshake: видео (9638) + ввод (9639) соединения открываются параллельно
3. Дорожка B (USB): сканирование устройств, USB ADB подключение (с диагностикой logSink), запуск приложения телефона
4. `checkAndAdvance()` оценивает состояние при изменении любого предусловия
5. Получает видео H.264 и отображает на экране автомобиля (ранний старт декодера на внеэкранной поверхности)
6. Захватывает события касаний и отправляет на телефон через соединение ввода для инжекции ввода VD

Состояния: IDLE → CONNECTING → CONNECTED → STREAMING

APK автомобиля встроен в APK телефона. Телефон автообновляет приложение авто через dadb при обнаружении несовпадения версий во время handshake. Автомобиль получает сообщение `UPDATING_CAR` и показывает статус вместо слепого переподключения.

### Двухрежимный UI

Приложение автомобиля разделяет процесс соединения и стриминг на два различных режима, соответствуя подходу приложения телефона:

- **Режим запуска** (`CarLaunchScreen`): Полноэкранный, с фокусом на соединение. Показывает брендинг, пошаговые инструкции по подключению, статус соединения и ручной ввод IP. Без панели навигации — весь экран посвящён установке соединения. Отображается при старте приложения авто и до получения значков приложений от телефона через управляющее соединение.

- **Режим стриминга** (`CarShell` с `PersistentNavBar`): Знакомый макет с левой панелью навигации (уведомления, домой, назад, недавние приложения) и областью контента (сетка приложений, зеркальный вид, уведомления). Отображается, когда `appList` не пуст и состояние CONNECTED или STREAMING.

Триггер перехода: когда телефон отправляет `APP_LIST` через управляющее соединение и состояние достигает CONNECTED/STREAMING, UI переключается с режима запуска на режим стриминга.

## Компоненты

### CarConnectionService

Служба переднего плана, управляющая полным жизненным циклом соединения с параллельным конечным автоматом по предусловиям и 3 выделенными соединениями.

**Архитектура 3 соединений:**
- `controlConnection`: handshake, heartbeat, команды приложений, канал DATA (список приложений, уведомления, логи авто)
- `videoConnection`: только видеокадры H.264 (телефон → авто)
- `inputConnection`: только события касаний (авто → телефон)
- Heartbeat/watchdog только на управляющем соединении; видео и ввод без накладных расходов heartbeat
- Отказ любого соединения каскадно → полный разрыв сессии

**Архитектура параллельных дорожек:**
- `connectionScope`: родительский Job для всех корутин обнаружения, отменяется при отключении
- Дорожки A и B работают одновременно
- `checkAndAdvance()` оценивает общее состояние при изменении любого предусловия
- Отключение пользователем: остаётся в IDLE, без автопереподключения (сохраняется в SharedPreferences)

**Дорожка A — WiFi:**
- Обнаружение: IP шлюза (hotspot/LAN, повтор каждые 3с) → mDNS → ручной IP
- Управляющее соединение: неблокирующее NIO SocketChannel подключение к телефону TCP:9637
- Handshake: отправляет размеры viewport + DPI + appVersionCode + appVersionName + targetFps (60) → получает информацию о телефоне + vdServerJarPath + connectionMethod
- При ответе handshake: открывает видео (9638) + ввод (9639) соединения параллельно, устанавливает `wifiReady = true` после установки всех 3
- Видео: получает кадры H.264 через видео соединение, направляет в VideoDecoder
- Касания: выделенный однопоточный исполнитель, `sendTouchEvent()` / `sendTouchBatch()` через соединение ввода
- Heartbeat: интервал 3с, таймаут watchdog 10с (только управляющее соединение)
- Backoff: экспоненциальная задержка при сбоях переподключения

**Дорожка B — USB:**
- Регистрирует `BroadcastReceiver` для `USB_DEVICE_ATTACHED` / `USB_DEVICE_DETACHED`
- `MainActivity` пересылает USB intent в службу
- Сканирует USB устройства на интерфейс ADB
- USB ADB подключение через UsbAdbConnection (в модуле protocol/), с `logSink`, направляющим все логи аутентификации ADB в FileLog телефона
- Запускает приложение телефона: `am start -n com.dilinkauto.client/.MainActivity`

**Поток обновления:**
- Если телефон отправляет `UPDATING_CAR`, авто устанавливает `updatingFromPhone = true`, показывает "Обновление приложения автомобиля..."
- Пропускает попытки видео/ввода и цикл переподключения во время обновления
- После `pm install -r`, приложение авто перезапускается заново

**Потоки состояний:**
- `_state`, `_phoneName`, `_appList`, `_notifications`, `_mediaMetadata`, `_playbackState`: основное состояние для UI
- `_videoReady`: true при получении первого не-CONFIG видеокадра
- `_statusMessage`: читаемый статус для отображения в UI
- `_vdStackEmpty` (SharedFlow): испускается, когда телефон сообщает об отсутствии активностей на VD (вызывает переход на домашний экран)

**Запуск сервера VD:**
- `deployVdServer()`: `shellNoWait` с CLASSPATH из vdServerJarPath handshake
- Аргументы: `W H DPI PORT EW EH FPS` — FPS из targetFps handshake

**Ранний старт декодера:**
- При первом CONFIG видеокадре запускает VideoDecoder на внеэкранном SurfaceTexture (до MirrorScreen)
- `onSurfaceTextureAvailable` MirrorScreen останавливает и перезапускает декодер с реальной поверхностью TextureView
- Предотвращает потерю ключевых кадров при задержке компоновки UI

**Маршрутизация логов авто:**
- `carLogSend()` направляет все логи со стороны авто через канал DATA `CAR_LOG` на телефон
- `videoDecoder.logSink` и `adb.setLogSink()` подключают логи VideoDecoder и UsbAdbConnection через тот же путь
- Все логи видны в `/sdcard/DiLinkAuto/client.log` телефона

### VideoDecoder

Декодер H.264 с использованием MediaCodec с выводом Surface (прямой рендеринг на GPU).

- Очередь кадров: 15 кадров — поглощает гонку при старте и сетевой джиттер
- `onFrameReceived()`: ставит кадры в очередь даже до вызова `start()`
- `start()`: сначала подаёт кешированный CONFIG, затем сливает очередь
- Удаление самого старого при переполнении очереди: предпочитает удалять P-кадры, вытесняет P-кадры из очереди для ключевых кадров/CONFIG
- `KEY_LOW_LATENCY = 1`, `KEY_PRIORITY = 0` для минимальной задержки декодирования
- CONFIG (SPS/PPS) кешируется и воспроизводится при перезапуске декодера
- Свойство `isRunning` для координации раннего старта
- Обратный вызов `logSink` направляет все логи декодера на телефон через carLogSend
- Режим catchup: четыре зоны градуированного ускорения на основе глубины очереди — нормальная (0-6 кадров), мягкая 1.5x (7-12 кадров, пропускает 1 из 3 неключевых), средняя 2x (13-20 кадров, пропускает 1 из 2), агрессивная 3x (21+ кадров, пропускает 2 из 3). Ключевые кадры никогда не пропускаются.
- Сбрасывает кодек и заново подаёт CONFIG при 10+ последовательных сбоях dequeueInputBuffer

### ServerApp

Класс приложения. Создаёт канал уведомлений `dilinkauto_car_service` с `IMPORTANCE_LOW`.

### RemoteAdbController

Прямой ADB клиент с использованием библиотеки dadb. Предоставляет касание, свайп, назад, домой и запуск приложений через shell команды на виртуальном дисплее. Используется как альтернативный путь ввода.

### CarLaunchScreen

Полноэкранный composable с фокусом на соединение, отображается до установки соединения с телефоном — без панели навигации, без сетки приложений.

- Брендинг DiLink Auto (иконка, заголовок, слоган)
- Карточка статуса соединения с цветным индикатором (зелёный=стриминг, оранжевый=подключено/подключение, серый=ожидание) и живым текстом статуса
- Инструкции "Как подключиться": 4 пронумерованных шага (включить hotspot, подключить USB, открыть приложение телефона, ждать автоподключения)
- Ручной ввод IP для прямого соединения
- Заменяется макетом режима стриминга, когда `appList` становится непустым и состояние достигает CONNECTED/STREAMING

### PersistentNavBar

Левая панель навигации 76dp — **только в режиме стриминга** — с:
- Отображением часов (HH:mm, обновление каждую секунду)
- Кнопкой отключения (отключает и сохраняет предпочтение пользователя)
- Индикатором состояния сети
- Кнопкой уведомлений со счётчиком непрочитанных
- Кнопкой домой
- Кнопкой назад
- Иконками недавних приложений (макс. 5, очищаются при недоступности)
- Иконки 40dp, текст 12-14sp

Ширина вычисляется для обеспечения чётного viewport для кодера H.264.

### NotificationScreen

- Список уведомлений, отсортированный по времени (новые сверху)
- Дедупликация по ID: обновления заменяют существующие (обработка прогресс-уведомлений)
- Индикаторы прогресса: определённые (заполненные) и неопределённые (вращающиеся)
- Запуск по нажатию: касание уведомления запускает приложение-владелец на VD и переключает на зеркальный вид

### Сетка приложений (HomeContent)

Отображается как основная область контента в режиме стриминга при текущем экране HOME:
- Поле поиска с `imePadding()` — клавиатура не сдвигает активность, только строка поиска
- `windowSoftInputMode="adjustNothing"` в манифесте
- Иконки приложений 64dp в динамически расчитываемых фиксированных колонках сетки (3-12 в зависимости от ширины дисплея)
- Текст имени приложения: bodyLarge
- Алфавитная сортировка
- Ручной ввод IP
- Статус соединения

### LauncherScreen (Устаревший)

Полный интегрированный макет запуска с `CarStatusBar`, `SideNavBar` (80dp) и `AppGrid`. Не используется в текущей маршрутизации `CarShell` — активный UI использует composable `PersistentNavBar` + `HomeContent`/`MirrorContent`/`NotificationContent` в строке.

### RecentAppsState

Отслеживает недавно запущенные приложения (макс. 5), сохраняемые в SharedPreferences. `pruneUnavailable()` удаляет приложения, которых больше нет при обновлении списка приложений.

### NavBarComponents

Отдельные composable виджеты панели навигации: `ClockDisplay` (обновление каждую секунду), `NetworkInfo` (состояние подключено/отключено), `RecentAppIcon` (40dp, с подсветкой активного состояния), `NavActionButton` (иконки 40dp, метки 12sp).

### CarTheme

Тёмная цветовая схема Material3 (`CarDark`) с цветами плиток приложений по категориям: Навигация (зелёный), Музыка (розовый), Связь (синий), Другое (серый).

### Информация о дисплее авто

Протестировано на BYD DiLink 3.0:
- Экран: 1920x990 @ 240dpi
- Viewport: ~1806x990 (после панели навигации 76dp)
- VD: ~3282x1800 @ 480dpi (GPU-масштабирование до ~1806x990 для кодирования)

## Зависимости

- Jetpack Compose + Material 3
- Модуль Protocol (общий с приложением телефона, включает UsbAdbConnection + AdbProtocol + VideoConfig)
- dadb 1.2.10 (резервный вариант TCP ADB)
