# Сервернае прыкладанне аўтамабіля (app-server)

## Агляд

Сервернае прыкладанне аўтамабіля працуе на інфармацыйна-забаўляльнай сістэме BYD DiLink. Яно выкарыстоўвае **паралельную мадэль злучэння** з адначасовай працай трэкаў WiFi (3 спецыялізаваныя злучэнні) і USB:

1. Track A (WiFi): gateway IP + mDNS discovery, control connect (9637), handshake з тэлефонам
2. Пасля handshake: відэа (9638) + уваходныя (9639) злучэнні адкрываюцца паралельна
3. Track B (USB): сканіраванне прылад, USB ADB connect (з logSink дыягностыкай), запуск прыкладання тэлефона
4. `checkAndAdvance()` ацэньвае стан пры змене любога прадумовы
5. Атрымлівае H.264 відэа і рэндэрыць на дысплеі аўтамабіля (ранні старт дэкодэра на пазаэкраннай паверхні)
6. Захоплівае падзеі дотыку і адпраўляе на тэлефон праз уваходнае злучэнне для ін'екцыі ўводу VD

Станы: IDLE → CONNECTING → CONNECTED → STREAMING

APK аўтамабіля ўбудаваны ў APK тэлефона. Тэлефон аўтаматычна абнаўляе прыкладанне аўтамабіля праз dadb пры выяўленні неадпаведнасці версій падчас handshake. Аўтамабіль атрымлівае паведамленне `UPDATING_CAR` і паказвае статус замест сляпога перападключэння.

### Двухрэжымны UI

Прыкладанне аўтамабіля падзяляе паток злучэння ад струменевага вопыту на два асобныя рэжымы, адпавядаючы падыходу прыкладання тэлефона:

- **Launch mode** (`CarLaunchScreen`): Поўнаэкранны, арыентаваны на злучэнне. Паказвае брэндынг, пакрокавыя інструкцыі па злучэнні, статус злучэння і ручны ўвод IP. Няма панэлі навігацыі — увесь экран прысвечаны ўсталяванню злучэння. Паказваецца пры запуску прыкладання аўтамабіля і застаецца, пакуль іконкі прыкладанняў не будуць атрыманы ад тэлефона праз control-злучэнне.

- **Streaming mode** (`CarShell` з `PersistentNavBar`): Знаёмы макет з левай панэллю навігацыі (апавяшчэнні, дадому, назад, нядаўнія прыкладанні) і вобласцю змесціва (сетка прыкладанняў, люстэркавы выгляд, апавяшчэнні). Паказваецца, калі `appList` не пусты і стан CONNECTED або STREAMING.

Трыгер пераходу: калі тэлефон адпраўляе `APP_LIST` праз control-злучэнне і стан злучэння дасягае CONNECTED/STREAMING, UI пераключаецца з рэжыму launch на рэжым streaming.

## Кампаненты

### CarConnectionService

Сэрвіс пярэдняга плана, які кіруе поўным жыццёвым цыклам злучэння з паралельным станавым аўтаматам перадумоў і 3 спецыялізаванымі злучэннямі.

**Архітэктура 3 злучэнняў:**
- `controlConnection`: handshake, heartbeat, каманды прыкладанняў, канал DATA (спіс прыкладанняў, апавяшчэнні, лагі аўтамабіля)
- `videoConnection`: толькі відэакадры H.264 (phone → car)
- `inputConnection`: толькі падзеі дотыку (car → phone)
- Heartbeat/watchdog толькі на control-злучэнні; відэа і ўваходныя не маюць накладных выдаткаў heartbeat
- Смерць любога злучэння каскадна → поўны разрыў сесіі

**Архітэктура паралельных трэкаў:**
- `connectionScope`: бацькоўскі Job для ўсіх каруцін discovery, адмяняецца пры адключэнні
- Track A і Track B працуюць адначасова
- `checkAndAdvance()` ацэньвае агульны стан пры змене любога прадумовы
- Адключэнне карыстальніка: застаецца IDLE, без аўтападключэння (захоўваецца ў SharedPreferences)

**Track A — WiFi:**
- Discovery: gateway IP (hotspot/LAN, паўтараецца кожныя 3s) → mDNS → manual IP
- Control connection: NIO неблакавальны SocketChannel connect да TCP тэлефона:9637
- Handshake: адпраўляе памеры вакна + DPI + appVersionCode + appVersionName + targetFps (60) → атрымлівае інфармацыю аб тэлефоне + vdServerJarPath + connectionMethod
- Пры адказе handshake: адкрывае відэа (9638) + уваходныя (9639) злучэнні паралельна, усталёўвае `wifiReady = true` пасля ўсталявання ўсіх 3
- Video: атрымлівае кадры H.264 праз відэа-злучэнне, адпраўляе ў VideoDecoder
- Touch: спецыялізаваны аднапатокавы выканаўца, `sendTouchEvent()` / `sendTouchBatch()` праз уваходнае злучэнне
- Heartbeat: інтэрвал 3s, тайм-аўт watchdog 10s (толькі control-злучэнне)
- Backoff: экспаненцыяльная затрымка пры збоях перападключэння

**Track B — USB:**
- Рэгіструе `BroadcastReceiver` для `USB_DEVICE_ATTACHED` / `USB_DEVICE_DETACHED`
- `MainActivity` перасылае USB intent у сэрвіс
- Сканіруе USB-прылады на інтэрфейс ADB
- USB ADB connect праз UsbAdbConnection (у модулі protocol/), з `logSink`, які маршрутызуе ўсе лагі аўтэнтыфікацыі ADB у FileLog тэлефона
- Запускае прыкладанне тэлефона: `am start -n com.dilinkauto.client/.MainActivity`

**Паток абнаўлення:**
- Калі тэлефон адпраўляе `UPDATING_CAR`, аўтамабіль усталёўвае `updatingFromPhone = true`, паказвае "Updating car app..." статус
- Прапускае спробы відэа/уваходных злучэнняў і цыкл перападключэння падчас абнаўлення
- Пасля `pm install -r`, прыкладанне аўтамабіля перазапускаецца нанова

**State Flows:**
- `_state`, `_phoneName`, `_appList`, `_notifications`, `_mediaMetadata`, `_playbackState`: асноўны стан для UI
- `_videoReady`: true, калі паступае першы неканфігурацыйны відэакадр
- `_statusMessage`: чытальны статус для адлюстравання ў UI
- `_vdStackEmpty` (SharedFlow): выпраменьваецца, калі тэлефон паведамляе, што VD не мае activity (запускае навігацыю дадому)

**Запуск VD-сервера:**
- `deployVdServer()`: `shellNoWait` з CLASSPATH з vdServerJarPath handshake
- Args: `W H DPI PORT EW EH FPS` — FPS перадаецца з targetFps handshake

**Ранні старт дэкодэра:**
- Пры першым відэакадры CONFIG запускае VideoDecoder на пазаэкранным SurfaceTexture (перад MirrorScreen)
- `onSurfaceTextureAvailable` MirrorScreen спыняе і перазапускае дэкодэр з рэальнай паверхняй TextureView
- Прадухіляе страту ключавых кадраў падчас затрымкі кампазіцыі UI

**Маршрутызацыя логаў аўтамабіля:**
- `carLogSend()` маршрутызуе ўсе лагі з боку аўтамабіля праз DATA-канал `CAR_LOG` на тэлефон
- `videoDecoder.logSink` і `adb.setLogSink()` злучаюць лагі VideoDecoder і UsbAdbConnection праз той жа шлях
- Усе лагі бачныя ў `/sdcard/DiLinkAuto/client.log` тэлефона

### VideoDecoder

Дэкодэр H.264 з выкарыстаннем MediaCodec з вывадам на Surface (прамое рэндэрынг GPU).

- Чарга кадраў: 15 кадраў — буферызуе гонку старту і сеткавы джытэр
- `onFrameReceived()`: ставіць кадры ў чаргу нават да выкліку `start()`
- `start()`: спачатку падае кэшаваны CONFIG, затым спусташае чаргу
- Выдаленне найстарэйшага пры запоўненай чарзе: аддае перавагу выдаленню P-кадраў, выцясняе P-кадры ў чарзе для ключавых кадраў/CONFIG
- `KEY_LOW_LATENCY = 1`, `KEY_PRIORITY = 0` для мінімальнай затрымкі дэкадавання
- CONFIG (SPS/PPS) кэшуецца і перайграецца пры перазапуску дэкодэра
- `isRunning` для каардынацыі ранняга старту
- Зваротны выклік `logSink` маршрутызуе ўсе лагі дэкодэра на тэлефон праз carLogSend
- Рэжым catchup: чатыры зоны паскарэння на аснове глыбіні чаргі — normal (0-6 кадраў), gentle 1.5x (7-12 кадраў, прапускае 1 з 3 неключавых), medium 2x (13-20 кадраў, прапускае 1 з 2), aggressive 3x (21+ кадраў, прапускае 2 з 3). Ключавыя кадры ніколі не прапускаюцца.
- Ачышчае кодэк і паўторна падае CONFIG пры 10+ паслядоўных скідах dequeueInputBuffer

### AppIconCache

Кэш іконак на баку аўтамабіля, які дэкадуе і змяняе памер іконак адзін раз, затым імгненна падае іх падчас пракруткі.

- `putSource(packageName, pngBytes)`: Захоўвае зыходны PNG высокага дазволу (192x192), атрыманы ад тэлефона праз спіс прыкладанняў. Захоўваецца на дыск як `{packageName}_src.png`.
- `prepareAll(apps, sizePx)`: Дэкадуе і змяняе памер усіх іконак у фонавым патоку перад тым, як сетка стане бачнай. Павялічвае `preparedVersion` для запуску рэкампазіцыі UI.
- `getPrepared(packageName)`: Сінхронны O(1) пошук ConcurrentHashMap — нуль каруцін, нуль I/O, нуль дэкадавання падчас пракруткі. Вяртае гатовы да рэндэрынгу `ImageBitmap`.
- `get(packageName, sizePx)`: Поўны шлях дэкадавання+змянення памеру (выкарыстоўваецца NotificationScreen і NavBar для некалькіх іконак). Вяртае `Bitmap` запытанага піксельнага памеру.
- `evict(packageName)`: Выдаляе кэшаваныя даныя для выдаленых прыкладанняў.

### ServerApp

Клас Application. Стварае канал апавяшчэнняў `dilinkauto_car_service` з `IMPORTANCE_LOW`.

### RemoteAdbController

Прамы кліент ADB з выкарыстаннем бібліятэкі dadb. Прадастаўляе tap, swipe, back, home і запуск прыкладанняў праз shell-каманды на віртуальным дысплеі. Выкарыстоўваецца як альтэрнатыўны шлях уводу.

### CarLaunchScreen

Поўнаэкранны кампазібл, арыентаваны на злучэнне, які паказваецца да ўсталявання злучэння з тэлефонам — без панэлі навігацыі, без сеткі прыкладанняў.

- Брэндынг DiLink Auto (іконка, загаловак, слоган)
- Картка статусу злучэння з каляровым індыкатарам (зялёны=streaming, аранжавы=connected/connecting, шэры=idle) і жывым тэкстам статусу
- Інструкцыі "How to connect": 4 нумараваныя крокі (уключыць hotspot, падключыць USB, адкрыць прыкладанне тэлефона, чакаць аўтападключэння)
- Ручны ўвод IP для прамога злучэння
- Замяняецца макетам рэжыму streaming, калі `appList` становіцца непустым і стан дасягае CONNECTED/STREAMING

### PersistentNavBar

76dp левая панэль навігацыі — **паказваецца толькі ў рэжыме streaming** — з:
- Гадзіннік (HH:mm, абнаўляецца кожную 1s)
- Кнопка Eject (адключае і захоўвае перавагу карыстальніка)
- Індыкатар стану сеткі
- Кнопка апавяшчэнняў з лічыльнікам непрачытаных
- Кнопка Home
- Кнопка Back
- Іконкі нядаўніх прыкладанняў (макс. 5, ачышчаюцца, калі прыкладанні становяцца недаступнымі)
- Іконкі 40dp, тэкст 12-14sp

Шырыня вылічваецца для гарантыі цотнага вакна для кадавальніка H.264.

### NotificationScreen

- Спіс апавяшчэнняў, адсартаваны па часе (спачатку новыя) з адносным адлюстраваннем часу ("now", "Xm", "Xh", дата)
- Іконкі прыкладанняў паказваюцца ўбудавана (з payload `iconPng` тэлефона)
- Дэдублікацыя па ID: абнаўленні замяняюць існуючыя (апрацоўвае апавяшчэнні з прагрэсам)
- Паласы прагрэсу: determinate (запоўненыя) і indeterminate (круцяцца)
- Запуск па націску: націск на апавяшчэнне запускае прыкладанне-ўладальнік на VD і пераключае на люстэркавы выгляд
- Кнопка закрыцця элемента (×): закрывае з анімацыяй slide-out, адпраўляе `NOTIFICATION_CLEAR` на тэлефон
- Кнопка "Clear All" у загалоўку: адпраўляе `NOTIFICATION_CLEAR_ALL` на тэлефон, ачышчае ўсе актыўныя апавяшчэнні

### Сетка прыкладанняў (HomeContent)

Паказваецца як асноўная вобласць змесціва, калі рэжым streaming актыўны і бягучы экран HOME:
- Поле пошуку з `imePadding()` — клавіятура не штурхае activity, рухаецца толькі радок пошуку
- `windowSoftInputMode="adjustNothing"` у manifest
- Іконкі прыкладанняў 64dp у дынамічна разлічваных фіксаваных калонках сеткі (3-12 у залежнасці ад шырыні дысплея)
- Тэкст назвы прыкладання: bodyLarge
- Алфавітнае сартаванне
- Доўгі націск для кантэкстнага меню: Выдаліць і Інфармацыя пра прыкладанне. Выдаленне прыкладання адпраўляе `APP_UNINSTALL` праз control-канал, тэлефон апрацоўвае сістэмны дыялог і адпраўляе `APP_UNINSTALLED` паведамленне даных для абнаўлення спісу прыкладанняў аўтамабіля. `AppInfoDataMessage` паказвае інфармацыю пра прыкладанне ў дыялогу на баку аўтамабіля.
- Ручны ўвод IP
- Статус злучэння

### LauncherScreen (састарэлы)

Поўны інтэграваны макет launcher з `CarStatusBar`, `SideNavBar` (80dp) і `AppGrid`. Не выкарыстоўваецца ў бягучай маршрутызацыі `CarShell` — актыўны UI выкарыстоўвае `PersistentNavBar` + `HomeContent`/`MirrorContent`/`NotificationContent` убудаваныя кампазіблы.

### RecentAppsState

Адсочвае нядаўна запушчаныя прыкладанні (макс. 5), захоўваецца ў SharedPreferences. `pruneUnavailable()` выдаляе прыкладанні, якія больш не прысутнічаюць пры абнаўленні спісу прыкладанняў.

### NavBarComponents

Асобныя кампазіблы віджэтаў панэлі навігацыі: `ClockDisplay` (абнаўляецца кожную 1s), `NetworkInfo` (стан connected/disconnected), `RecentAppIcon` (40dp, з падсветкай актыўнага стану), `NavActionButton` (іконкі 40dp, меткі 12sp).

### CarTheme

Цёмная каляровая схема Material3 (`CarDark`) з катэгарыйнымі колерамі плітак прыкладанняў: Navigation (зялёны), Music (ружовы), Communication (сіні), Other (шэры).

### Інфармацыя аб дысплеі аўтамабіля

Пратэставана на BYD DiLink 3.0:
- Экран: 1920x990 @ 240dpi
- Вакно: ~1806x990 (пасля 76dp панэлі навігацыі)
- VD: ~3282x1800 @ 480dpi (GPU-маштабавана да ~1806x990 для кадавання)

## Залежнасці

- Jetpack Compose + Material 3
- Protocol module (агульны з прыкладаннем тэлефона, уключае UsbAdbConnection + AdbProtocol + VideoConfig)
- dadb 1.2.10 (TCP ADB рэзервовы варыянт)
